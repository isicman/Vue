    # 【题目2】Volume-emptyDir

    使用 image: redis:alpine 创建一个名为 redis-storage 的 Pod，其类型为 emptyDir 的 Volume 持续 Pod 的生命周期。 规格如下。

    使用命令 kubectl run 并为 redis-storage pod 创建 pod 定义文件并添加卷。

    运行以下命令： kubectl run redis-storage --image=redis:alpine --dry-run=client -o yaml > redis-storage.yaml 并在其中添加卷 emptyDir 。

    • Pod named 'redis-storage' created
    • Pod 'redis-storage' uses Volume type of emptyDir
    • Pod 'redis-storage' uses volumeMount with mountPath = /data/redis
    ---
apiVersion: v1
kind: Pod
metadata:
  creationTimestamp: null
  labels:
    run: redis-storage
  name: redis-storage
spec:
  containers:
  - image: redis:alpine
    name: redis-storage
    volumeMounts:
    - mountPath: /data/redis
      name: temp-volume
  volumes:
  - name: temp-volume
    emptyDir: {}

    # 【题目4】PVC

    在 /root/CKA/use-pv.yaml 中创建了一个 pod 定义文件。使用此清单文件并挂载名为 pv-1 的持久卷。确保 pod 正在运行并且 PV 已绑定。

    挂载路径：/data

    持久卷声明名称：my-pvc。

    请求容量： 10Mi

    • persistentVolume Claim configured correctly
    • pod using the correct mountPath
    • pod using the persistent volume claim
    ---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
spec:
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
       storage: 10Mi
---
apiVersion: v1
kind: Pod
metadata:
  creationTimestamp: null
  labels:
    run: use-pv
  name: use-pv
spec:
  containers:
  - image: nginx
    name: use-pv
    volumeMounts:
    - mountPath: "/data"
      name: mypd
  volumes:
    - name: mypd
      persistentVolumeClaim:
        claimName: my-pvc

    # 【题目5】Deployment

    创建一个名为 nginx-deploy 的新部署，使用镜像 nginx:1.16 和 1 个副本。接下来使用滚动更新将部署升级到版本 1.17。

    • Deployment : nginx-deploy. Image: nginx:1.16
    • Image: nginx:1.16
    • Task: Upgrade the version of the deployment to 1:17
    • Task: Record the changes for the image upgrade
    $ kubectl create deployment nginx-deploy --image=nginx:1.16 --dry-run=client -o yaml > deploy.yaml
    1

    To create a resource from definition file and to record:

    $ kubectl apply -f deploy.yaml --record
    1

    To view the history of deployment nginx-deploy:

    $ kubectl rollout history deployment nginx-deploy
    1

    To upgrade the image to next given version:

    $ kubectl set image deployment/nginx-deploy nginx=nginx:1.17 --record
    1

    To view the history of deployment nginx-deploy:

    $ kubectl rollout history deployment nginx-deploy
    1

    • Pod: nginx-resolver created
    • Service DNS Resolution recorded correctly
    • Pod DNS resolution recorded correctly

    To create a pod nginx-resolver and expose it internally:

    # 【题目8】静态Pod

    在 node01 上创建一个名为 nginx-critical 的静态 pod，使用 nginx 映像，并确保它在出现故障时自动重新创建/重新启动。 例如，使用 /etc/kubernetes/manifests 作为静态 Pod 路径。

    • static pod configured under /etc/kubernetes/manifests ?
    • Pod nginx-critical-node01 is up and running

    To create a static pod called nginx-critical by using below command:

    $ kubectl run nginx-critical --image=nginx --dry-run=client -o yaml > static.yaml
    1

    Copy the contents of this file or use scp command to transfer this file from controlplane to node01 node.

    root@controlplane:~$ scp static.yaml node01:/root/
    1

    To know the IP Address of the node01 node:

    root@controlplane:~$ kubectl get nodes -o wide

# Perform SSH
root@controlplane:~$ ssh node01
OR
root@controlplane:~$ ssh <IP of node01>
    1
    2
    3
    4
    5
    6

    On node01 node: Check if static pod directory is present which is /etc/kubernetes/manifests, if it's not present then create it.

    root@node01:~$ mkdir -p /etc/kubernetes/manifests
    1

    Add that complete path to the staticPodPath field in the kubelet config.yaml file.

    root@node01:~$ vi /var/lib/kubelet/config.yaml

    # 【题目9】NetworkPolicy

    创建网络策略以仅允许从内部应用程序到 payroll-service的name=internal和 db-service的name=payroll流量。

    • Policy Name: internal-policy
    • Policy Type: Egress
    • Egress Allow: payroll
    • Payroll Port: 8080
    • Egress Allow: mysql
    • MySQL Port: 3306
    apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: internal-policy
  namespace: default
spec:
  podSelector:
    matchLabels:
      name: internal
  policyTypes:
  - Egress
  - Ingress
  ingress:
    - {}
  egress:
  - to:
    - podSelector:
        matchLabels:
          name: mysql
    ports:
    - protocol: TCP
      port: 3306

  - to:
    - podSelector:
        matchLabels:
          name: payroll
    ports:
    - protocol: TCP
      port: 8080

  - ports:
    - port: 53
      protocol: UDP
    - port: 53
      protocol: TCP
